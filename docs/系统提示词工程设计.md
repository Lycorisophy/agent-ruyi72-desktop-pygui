# DeerFlow 提示词工程设计总结

## 概述

DeerFlow 的提示词工程采用**分层模板 + 动态组合**的设计理念，通过模块化的提示词组件实现不同代理的灵活配置。系统支持主代理（Lead Agent）、记忆代理（Memory Agent）和子代理（Subagent）三种角色的提示词定制。

---

## 一、主代理提示词工程

### 1.1 系统提示词模板结构

主代理的提示词通过 `SYSTEM_PROMPT_TEMPLATE` 模板定义，包含以下关键区块：

```
<role>                        # 代理身份定义
<soul>                        # Agent Soul (SOUL.md) 个性化
<memory>                      # 记忆上下文注入
<thinking_style>              # 思考风格指导
<clarification_system>        # 需求澄清系统
<skill_system>                # 技能系统
<available-deferred-tools>    # 延迟工具列表
<subagent_system>             # 子代理系统
<working_directory>           # 工作目录配置
<response_style>              # 响应风格规范
<citations>                   # 引用规范
<critical_reminders>          # 关键提醒
<current_date>                # 当前日期
```

### 1.2 核心提示词组件

#### 角色定义 (Role)
```python
"You are {agent_name}, an open-source super agent."
```
动态注入代理名称，支持多代理实例化。

#### 代理灵魂 (Agent Soul)
从 `{base_dir}/agents/{agent_name}/SOUL.md` 文件加载个性化配置，支持每个代理拥有独特的行为风格。

#### 记忆上下文 (Memory Context)
```python
<memory>
  User Context:
  - Work: ...
  - Personal: ...
  - Current Focus: ...

  History:
  - Recent: ...
  - Earlier: ...

  Facts:
  - [knowledge | 0.95] ...
</memory>
```
基于 v3.0 记忆系统的结构化用户信息注入，使用 tiktoken 精确控制 token 预算。

#### 思考风格指导 (Thinking Style)
```markdown
- Think concisely and strategically about the user's request BEFORE taking action
- Break down the task: What is clear? What is ambiguous? What is missing?
- **PRIORITY CHECK: If anything is unclear, missing, or has multiple interpretations, 
  you MUST ask for clarification FIRST**
- Never write down your full final answer in thinking, only outline
- After thinking, you MUST provide your actual response
```

#### 需求澄清系统 (Clarification System)
**强制性澄清场景**：
1. **缺失信息** (`missing_info`): 缺少必要细节
2. **模糊需求** (`ambiguous_requirement`): 存在多种合理解释
3. **方案选择** (`approach_choice`): 多种有效方案可选
4. **风险操作** (`risk_confirmation`): 破坏性操作需确认
5. **建议提案** (`suggestion`): 推荐方案需审批

**执行原则**：`CLARIFY → PLAN → ACT`，澄清必须先于行动。

#### 子代理编排系统 (Subagent System)
```markdown
**🚀 SUBAGENT MODE ACTIVE - DECOMPOSE, DELEGATE, SYNTHESIZE**

**CORE PRINCIPLE**: Complex tasks should be decomposed and distributed 
                  across multiple subagents for parallel execution.

**⛔ HARD CONCURRENCY LIMIT: MAXIMUM {n} `task` CALLS PER RESPONSE.**

**DECOMPOSE + PARALLEL EXECUTION (Preferred)**:
- Complex research questions
- Multi-aspect analysis
- Large codebases analysis
- Comprehensive investigations

**DO NOT use subagents (execute directly)**:
- Cannot be decomposed
- Ultra-simple actions
- Need immediate clarification
- Sequential dependencies
```

#### 工作目录配置 (Working Directory)
```markdown
<working_directory>
- User uploads: `/mnt/user-data/uploads`
- User workspace: `/mnt/user-data/workspace`
- Output files: `/mnt/user-data/outputs`
</working_directory>
```
沙箱环境的虚拟路径映射配置。

#### 引用规范 (Citations)
```markdown
**CRITICAL: Always include citations when using web search results**

- When to Use: MANDATORY after web_search, web_fetch
- Format: `[citation:TITLE](URL)` inline
- Sources Section: Full reference list at end
- Format in Sources: `[Title](URL) - Description`
```

### 2.1 记忆更新提示词 (MEMORY_UPDATE_PROMPT)

```markdown
Current Memory State:
<current_memory>{current_memory}</current_memory>

New Conversation to Process:
<conversation>{conversation}</conversation>

Instructions:
1. Analyze the conversation for important information
2. Extract relevant facts, preferences, context
3. Update memory sections following length guidelines
```

## 三、子代理提示词工程

### 3.1 General-Purpose 子代理

```markdown
You are a general-purpose subagent working on a delegated task. 
Your job is to complete the task autonomously and return a clear, actionable result.

<guidelines>
- Focus on completing the delegated task efficiently
- Use available tools as needed
- Think step by step but act decisively
- Return a concise summary
- Do NOT ask for clarification - work with the information provided
</guidelines>

<output_format>
1. A brief summary of what was accomplished
2. Key findings or results
3. Relevant file paths, data, or artifacts created
4. Issues encountered (if any)
5. Citations: Use `[citation:Title](URL)` format
</output_format>
```

**工具限制**：
- `tools=None`: 继承父代理所有工具
- `disallowed_tools=["task", "ask_clarification", "present_files"]`: 禁止嵌套

## 四、提示词工程最佳实践

### 4.1 分层设计原则

| 层级 | 内容 | 变更频率 |
|------|------|----------|
| 静态层 | 角色定义、响应风格、引用规范 | 低 |
| 半静态层 | 思考风格、澄清系统、工作目录 | 中 |
| 动态层 | 记忆上下文、技能列表、子代理配置 | 高 |

### 4.2 Token 预算控制

- 记忆注入：默认 2000 tokens，使用 tiktoken 精确计算
- 技能区块：按需加载，只注入已启用的技能
- 延迟工具：仅列出工具名称，实际工具按需加载

### 4.3 渐进式加载模式

```markdown
**Progressive Loading Pattern:**
1. When a user query matches a skill's use case, immediately call `read_file` 
   on the skill's main file
2. Read and understand the skill's workflow
3. Load referenced resources only when needed
4. Follow the skill's instructions precisely
```

## 总结

DeerFlow 的提示词工程具有以下特点：

1. **模块化设计**：提示词由独立的功能区块组成，便于组合和定制
2. **动态组装**：运行时根据配置和上下文动态生成最终提示词
3. **层次分明**：静态/半静态/动态分层，平衡灵活性和性能
4. **Token 精确控制**：使用 tiktoken 精确计算，避免超出模型限制
5. **多代理支持**：通过 agent_name 参数支持多个代理实例
