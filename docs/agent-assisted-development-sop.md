# Agent 辅助开发标准作业程序（SOP）

> **目的**：在 AI Agent 参与需求、设计、实现、测试与闭环时，统一阶段划分、产出物与反馈路径，减少遗漏与返工。  
> **适用**：本仓库及同类桌面/服务端项目的迭代；具体工具（Cursor、CI、测试框架）以实现为准。

---

## 总览流程

```text
[阶段一] DST 建立用户画像与故事
    ↓
[阶段二] 五步法建立 PRD（含 .feature 实例）
    ↓
[阶段三] 五步法对齐设计文档（含 YAML/DDL/时序图）
    ↓
[阶段三-补充] Agent 辅助开发实现（脚手架生成 + 单测 + 预检）  ← 新增
    ↓
[阶段四] Agent 辅助智能测试（自动化验收与回归）
    ↓
[阶段五] Agent 辅助问题修复（根因诊断与修复建议）
    ↓
[阶段六] Agent 辅助问题闭环（门禁与知识沉淀）
    ↓
(反馈至阶段二) 更新 PRD 实例化用例库
```

---

## 阶段一：DST（对话状态追踪）建立用户画像与故事

**目标**：在写需求前，用结构化对话沉淀「谁、在什么场景、要什么、痛点是什么」。

| 项 | 说明 |
|----|------|
| **用户画像** | 角色、技能水平、使用环境、频率与目标（可多条 persona） |
| **故事（Story）** | 用户旅程：触发情境 → 行为 → 期望结果 → 当前阻碍 |
| **DST 要点** | 维护会话内关键槽位（主题、优先级、约束），避免需求在对话中漂移 |

**产出物**：用户画像摘要、1～3 条核心用户故事（可对应后续 PRD 中的实例化场景）。

**出口准则**：故事可被非技术人员读懂，且与业务目标对齐。

---

## 阶段二：五步法建立 PRD（含 .feature 实例）

五步法用于把模糊故事变成**可评审、可测**的产品需求。

| 步骤 | 内容 | 产出提示 |
|------|------|----------|
| **1. 主动定义问题** | 问题陈述、非目标（Out of scope）、成功/失败边界 | 一句话问题定义 + 范围声明 |
| **2. 明确数据维度** | 输入/输出实体、来源、更新频率、隐私与合规 | 数据清单表（字段级可后置到阶段三） |
| **3. 设计科学的衡量指标** | 北极星指标、过程指标、防虚荣指标说明 | 指标名、口径、采集方式 |
| **4. 对齐决策导向** | 关键决策点、备选方案、取舍理由 | ADR 简表或 PRD 内「决策」小节 |
| **5. 明确验收标准、实例化需求** | 可测试的 DoD、Given-When-Then | **`.feature` 或实例表**：每条对应可自动化或人工走查的用例 |

**产出物**：PRD 主文档 + **`.feature`（或等价实例文件）**，作为阶段四测试与门禁的输入。

**出口准则**：每条 P0/P1 需求至少有一条实例化用例可追溯。

---

## 阶段三：五步法对齐设计文档（含 YAML/DDL/时序图）

将 PRD「落地」为研发可执行的设计，五步法与 PRD 五步法**对应但偏工程**。

| 步骤 | 内容 | 产出提示 |
|------|------|----------|
| **1. 业务场景拆解为系统时序** | 参与者、同步/异步边界、失败重试 | **时序图**（Mermaid/PlantUML 等） |
| **2. 功能点映射为接口定义** | API、事件、CLI；错误码与幂等 | **OpenAPI/YAML** 或等价接口清单 |
| **3. 实体属性收敛为数据模型** | 表、索引、迁移策略 | **DDL** 或 ORM 模型 + 迁移脚本说明 |
| **4. 非功能需求量化为架构决策** | 性能、可用性、安全、可观测性 | **架构决策记录**：指标阈值与实现抓手 |
| **5. 设计文档对齐检查清单** | PRD 实例是否均有设计覆盖、接口与数据是否闭环 | **检查清单**（表格打勾） |

**产出物**：设计说明（可拆为多篇）+ **YAML（接口）** + **DDL（数据）** + **时序图** + 对齐清单。

**出口准则**：阶段二中的 `.feature` 能在设计中找到接口/数据/时序支撑，无未映射的 P0 需求。

---

## 阶段三-补充：Agent 辅助开发实现（新增）

在设计冻结后、大规模联调前，由 Agent 辅助**加速实现与质量基线**。

| 活动 | 说明 |
|------|------|
| **脚手架生成** | 按模块/接口生成目录、桩代码、配置占位，遵循仓库既有风格 |
| **单测** | 为核心逻辑与边界条件补充单元测试；与 `.feature` 关键路径对齐 |
| **预检** | 静态检查、编译、最小化冒烟脚本；必要时生成 PR 自检清单 |

**产出物**：可合并的代码分支 + 测试 + 预检通过记录。

**出口准则**：CI 或本地预检通过；P0 用例至少有自动化覆盖或明确人工步骤。

---

## 阶段四：Agent 辅助智能测试

**目标**：自动化验收与回归，持续对齐 PRD 实例与设计。

| 项 | 说明 |
|----|------|
| 用例来源 | 阶段二 `.feature` + 阶段三接口与数据约束 |
| 策略 | 优先级分层（冒烟 / 全量 /  nightly）；测试数据与环境隔离 |
| Agent 角色 | 生成/补全用例、解析失败日志、辅助定位 flaky |

**产出物**：测试报告、缺陷列表、覆盖率或等价质量信号（按项目约定）。

---

## 阶段五：Agent 辅助问题修复

**目标**：缩短从「失败现象」到「可验证修复」的路径。

| 项 | 说明 |
|----|------|
| 根因诊断 | 复现步骤、日志与版本、假设树（环境 / 代码 / 数据） |
| 修复建议 | 最小变更、副作用、回滚方式 |
| Agent 角色 | 辅助对比 diff、检索相似历史问题、草拟补丁与说明 |

**产出物**：修复 PR 或补丁 + 根因简述（可进阶段六知识库）。

---

## 阶段六：Agent 辅助问题闭环

**目标**：门禁通过且经验可复用。

| 项 | 说明 |
|----|------|
| 门禁 | 合并条件：测试通过、评审、发布窗口等（按团队规范） |
| 知识沉淀 | 故障模式、规避方式、文档或 Runbook 更新 |
| Agent 角色 | 起草发布说明、更新 FAQ/排障文档、建议 PRD/设计增补点 |

**产出物**：闭环记录（可工单）+ 可选文档更新。

---

## 反馈闭环：回到阶段二

当阶段四～六发现**需求歧义、验收不足或设计缺口**时：

1. 在阶段二**更新 PRD** 与 **实例化用例库**（`.feature` 或等价表）。  
2. 若涉及接口或数据契约，同步触发阶段三**增量对齐**（小版本设计变更）。  
3. 重大方向变化应重新走阶段一～二对齐，避免仅在实现层打补丁。
