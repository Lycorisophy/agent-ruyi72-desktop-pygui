# DeerFlow 约束工程设计总结

## 概述

DeerFlow 的约束工程采用**多层防护 + 中间件编排**的设计理念，通过沙箱隔离、权限控制、行为约束和资源限制四层机制确保 Agent 安全可控地执行任务。约束系统完全通过中间件实现，无需修改核心代理逻辑。

---

## 一、沙箱机制 (Sandbox)

### 1.1 沙箱架构

```
┌─────────────────────────────────────────────────────────────┐
│                      Host (Backend)                         │

│  │              SandboxMiddleware                        │   │
│  │  ┌─────────────────────────────────────────────────┐ │   │

│  │  │  ┌──────────────┐  ┌──────────────────────────┐│ │   │
│  │  │  │LocalSandbox  │  │   AioSandboxProvider     ││ │   │
│  │  │  │  (直接执行)   │  │   (Docker 容器隔离)       ││ │   │
│  │  │  └──────────────┘  └──────────────────────────┘│ │   │
│  │  └─────────────────────────────────────────────────┘ │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ Volume Mount
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              Container (Sandbox Environment)                │
│                                                             │
│  /mnt/user-data/                                            │
│    ├── workspace/    (可写工作目录)                          │
│    ├── uploads/     (用户上传文件)                           │
│    └── outputs/     (最终交付物)                            │
│                                                             │
│  /mnt/skills/                                              │
│    ├── safe/        (内置安全技能)                          │
│    └── custom/      (用户自定义技能)                        │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 沙箱提供器

#### LocalSandboxProvider
- 直接在宿主机执行命令（危险，仅用于完全可信的本地工作流）
- 配置选项 `allow_host_bash=True` 可启用

#### AioSandboxProvider
- Docker 容器隔离执行
- 可配置项：
  - `image`: Docker 镜像
  - `port`: 基础端口
  - `replicas`: 最大并发容器数（默认 3）
  - `container_prefix`: 容器名称前缀
  - `idle_timeout`: 空闲超时秒数（默认 600 = 10分钟）
  - `mounts`: 主机到容器的路径映射
  - `environment`: 环境变量注入

### 1.3 生命周期管理

```python
class SandboxMiddleware:
    def __init__(self, lazy_init: bool = True):
        """
        lazy_init=True:  首次工具调用时获取沙箱
        lazy_init=False: 主代理调用前立即获取沙箱
        """
        
    def before_agent(self, state, runtime):
        if self._lazy_init:
            return  # 延迟到工具调用
        sandbox_id = self._acquire_sandbox(thread_id)
        return {"sandbox": {"sandbox_id": sandbox_id}}
    
    def after_agent(self, state, runtime):
        get_sandbox_provider().release(sandbox_id)
```

**复用策略**：沙箱在同一个线程内跨多轮复用，避免重复创建开销。清理在应用关闭时通过 `SandboxProvider.shutdown()` 完成。

## 二、安全约束 (Security Constraints)

### 2.1 Bash 命令审计 (SandboxAuditMiddleware)

```python
_HIGH_RISK_PATTERNS = [
    r"rm\s+-[^\s]*r[^\s]*\s+(/\*?|~/?\*?|/home\b|/root\b)",  # rm -rf / * ~ /home /root
    r"(curl|wget).+\|\s*(ba)?sh",  # curl|sh, wget|sh
    r"dd\s+if=",  # dd 命令
    r"mkfs",  # 格式化文件系统
    r"cat\s+/etc/shadow",  # 读取密码文件
    r">\s*/etc/",  # 覆盖系统文件
]

_MEDIUM_RISK_PATTERNS = [
    r"chmod\s+777",  # 过度宽松权限
    r"pip\s+install",  # 安装包
    r"pip3\s+install",
    r"apt(-get)?\s+install",
]
```

**处理策略**：
| 风险等级 | 命令示例 | 处理方式 |
|----------|----------|----------|
| 高危 (block) | `rm -rf /`, `curl url \| bash` | 阻止执行，返回错误消息 |
| 中危 (warn) | `pip install`, `chmod 777` | 执行但追加警告 |

**审计日志**：
```json
{
  "timestamp": "2026-04-13T10:30:00Z",
  "thread_id": "abc123",
  "command": "pip install requests",
  "verdict": "warn"
}
```

### 2.2 Guardrails 预检机制 (GuardrailMiddleware)

```python
class GuardrailsConfig(BaseModel):
    enabled: bool = False
    fail_closed: bool = True  # 提供者错误时默认阻止
    passport: str | None = None  # OAP passport 路径
    provider: GuardrailProviderConfig | None = None

class GuardrailMiddleware:
    def wrap_tool_call(self, request, handler):
        decision = self.provider.evaluate(GuardrailRequest(
            tool_name=tool_name,
            tool_input=tool_args,
            agent_id=self.passport,
            timestamp=datetime.now(UTC).isoformat(),
        ))
        if not decision.allow:
            return ToolMessage(
                content=f"Guardrail denied: tool '{tool_name}' was blocked. Reason: {reason}",
                status="error",
            )
        return handler(request)
```

**决策语义**：
- `allow=True`: 执行工具
- `allow=False`: 返回错误消息，代理可适应
- `fail_closed=True`: 提供者异常时阻止调用

**检测算法**：
```python
def _hash_tool_calls(tool_calls: list[dict]) -> str:
    # 规范化工具调用（去序）
    normalized = [{"name": tc.get("name"), "args": tc.get("args")} for tc in tool_calls]
    # 排序确保相同调用集产生相同哈希
    normalized.sort(key=lambda tc: (tc["name"], json.dumps(tc["args"], sort_keys=True)))
    blob = json.dumps(normalized, sort_keys=True)
    return hashlib.md5(blob.encode()).hexdigest()[:12]
```

**处理流程**：
```
检测 → 计数 → 判断阈值
         ↓
    warn_threshold(3) → 注入 HumanMessage 警告
         ↓
    hard_limit(5) → 剥离 tool_calls，强制输出文本
```

**执行逻辑**：
```python
def _truncate_task_calls(self, state: AgentState) -> dict | None:
    tool_calls = last_msg.tool_calls
    task_indices = [i for i, tc in enumerate(tool_calls) if tc.get("name") == "task"]
    if len(task_indices) > self.max_concurrent:
        indices_to_drop = set(task_indices[self.max_concurrent:])
        truncated_tool_calls = [tc for i, tc in enumerate(tool_calls) if i not in indices_to_drop]
        return {"messages": [last_msg.model_copy(update={"tool_calls": truncated_tool_calls})]}
```

**多批次编排**（由提示词引导）：
```markdown
**MULTI-BATCH EXECUTION** (>max_concurrent sub-tasks):
Turn 1: Launch sub-tasks 1-{n} in parallel
Turn 2: Launch next batch in parallel
... continue until all sub-tasks are complete
Final turn: Synthesize ALL results
```

## 四、资源限制 (Resource Limits)

### 4.1 工作空间隔离

```python
class Paths:
    def thread_dir(self, thread_id: str) -> Path:
        # 路径安全验证
        if not _SAFE_THREAD_ID_RE.match(thread_id):
            raise ValueError(f"Invalid thread_id: only alphanumeric, hyphens, underscores allowed")
        return self.base_dir / "threads" / thread_id

    def sandbox_work_dir(self, thread_id: str) -> Path:
        """Host: {base_dir}/threads/{thread_id}/user-data/workspace/
           Sandbox: /mnt/user-data/workspace/"""
        return self.thread_dir(thread_id) / "user-data" / "workspace"

    def sandbox_uploads_dir(self, thread_id: str) -> Path:
        """Host: {base_dir}/threads/{thread_id}/user-data/uploads/
           Sandbox: /mnt/user-data/uploads/"""

    def sandbox_outputs_dir(self, thread_id: str) -> Path:
        """Host: {base_dir}/threads/{thread_id}/user-data/outputs/
           Sandbox: /mnt/user-data/outputs/"""
```

**虚拟路径映射**：
| 宿主机路径 | 沙箱内路径 | 用途 |
|-----------|-----------|------|
| `.../threads/{thread}/user-data/workspace/` | `/mnt/user-data/workspace/` | 临时工作文件 |
| `.../threads/{thread}/user-data/uploads/` | `/mnt/user-data/uploads/` | 用户上传文件 |
| `.../threads/{thread}/user-data/outputs/` | `/mnt/user-data/outputs/` | 最终交付物 |
| `.../threads/{thread}/acp-workspace/` | `/mnt/acp-workspace/` | ACP 代理工作区 |

**路径遍历防护**：
```python
def resolve_virtual_path(self, thread_id: str, virtual_path: str) -> Path:
    stripped = virtual_path.lstrip("/")
    prefix = "mnt/user-data"
    if not stripped.startswith(prefix + "/"):
        raise ValueError("Path must start with /mnt/user-data")
    
    relative = stripped[len(prefix):].lstrip("/")
    base = self.sandbox_user_data_dir(thread_id).resolve()
    actual = (base / relative).resolve()
    
    try:
        actual.relative_to(base)
    except ValueError:
        raise ValueError("Access denied: path traversal detected")
```

### 5.2 中间件插入机制

```python
def _insert_extra(chain, extras):
    # 支持 @Next/@Prev 锚点定位
    for mw in extras:
        next_anchor = getattr(type(mw), "_next_anchor", None)
        prev_anchor = getattr(type(mw), "_prev_anchor", None)
        
        if next_anchor:
            idx = chain.index(next_anchor)
            chain.insert(idx + 1, mw)
        elif prev_anchor:
            idx = chain.index(prev_anchor)
            chain.insert(idx, mw)
```

## 六、约束工程最佳实践

### 6.1 分层防护原则

```
第一层：输入约束（沙箱隔离）
  └─ 子代理独立工作空间
  └─ 路径遍历防护

第二层：执行约束（行为限制）
  └─ 高危命令阻断
  └─ 循环检测

第三层：后验约束（资源管理）
  └─ Token 使用统计
  └─ 超时控制
```

### 6.2 失败策略

| 组件 | 默认策略 | 说明 |
|------|----------|------|
| GuardrailMiddleware | fail_closed | 提供者错误时阻止调用 |
| SandboxAuditMiddleware | block high, warn medium | 高危阻断，中危警告 |
| LoopDetectionMiddleware | warn → hard_stop | 软警告，硬停止 |
| SubagentLimitMiddleware | truncate | 截断超出的调用 |

### 6.3 可观测性

- 所有安全决策记录审计日志
- 中间件状态可通过 reset() 方法清除
- 线程级隔离避免状态污染

## 总结

DeerFlow 的约束工程具有以下特点：

1. **中间件驱动**：所有约束通过 LangGraph 中间件实现，不修改核心代理逻辑
2. **沙箱隔离**：Docker 容器或本地隔离执行，防止系统级损害
3. **多层防护**：从输入到执行到后验的多层约束机制
4. **智能决策**：基于规则和模式匹配的自动决策，区分阻塞/警告
5. **可配置性**：通过配置文件灵活启用/禁用各层约束
6. **线程隔离**：每个会话独立的工作空间，支持并发安全
