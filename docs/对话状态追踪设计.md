# 对话状态追踪设计（全模式）

## 1. 目标与范围

**目标**：在单一会话内，对「当前对话进行到哪一步、是否占用 LLM、是否可打断、本轮是否已落盘」形成**可查询、可持久化（可选）、可观测**的统一模型；**所有用户可见模式**共用同一套顶层语义，仅在 **mode × session_variant** 下挂载不同子状态。

**覆盖矩阵**（与 [`SessionMeta`](../src/storage/session_store.py) 一致）：

| `session_variant` | 允许的 `mode` | 主要执行路径 |
|-------------------|---------------|--------------|
| `standard` | `chat` / `react` / `persona` | [`SafeChatStreamRuntime`](../src/agent/chat_stream_runtime.py) / [`run_react`](../src/agent/react_lc.py) 后台线程 / [`PersonaRuntime`](../src/agent/persona_runtime.py) |
| `team` | 仅 `chat`（侧栏约束） | [`run_team_turn`](../src/agent/team_turn.py)，同步一轮多槽模型 |
| `knowledge` | `chat` / `react`（无拟人） | 同上 Chat/ReAct，附加知识库 system |

**不在本文一次性落地全部代码**：下文给出**数据模型、状态机、事件映射与分阶段实现**；与现有 `turn_id`、`persona_emit` / `react_stream_emit`、`llm_busy` **对齐而非推翻**。

---

## 2. 现状资产（实现设计的锚点）

| 机制 | 位置 | 作用 |
|------|------|------|
| **`turn_id`** | `SafeChatStreamRuntime`、`PersonaRuntime` | 每轮用户发送递增，事件里携带，用于前端区分流式片段归属 |
| **事件协议** | `turn.started` / `token.delta` / `turn.completed` / `turn.cancelled` / `turn.finished` / `message.final` / `error` | Chat 与拟人已对齐；ReAct 完成时亦发 `turn.finished`（见 [`ConversationService`](../src/service/conversation.py)） |
| **`llm_busy`** | `ConversationService.llm_busy()` | 标记本进程正占用 LLM（对话 / ReAct / 记忆抽取等），供闲时任务避让 |
| **消息持久化** | `SessionStore.save_messages` | 用户/助手消息与可选 `card`；**不包含**细粒度「进行中」状态 |

**缺口**：缺少会话级 **单一真源** 的 `phase`（及可选子状态），重启后 UI 无法从磁盘恢复「上一轮是否未结束」；ReAct **步级**状态仅能通过流式回调间接感知，未与顶层模型统一。

## 3. 统一状态模型

### 3.1 顶层 `DialoguePhase`（建议枚举）

语义上覆盖全模式；同一时刻会话只有一个主相位（并可有**叠加标记**，见 3.3）。

| 值 | 含义 |
|----|------|
| `idle` | 无进行中的模型轮次，等待用户输入 |
| `streaming` | 流式生成中（Chat 或拟人：`SafeChatStreamRuntime` / `PersonaRuntime`） |
| `react_running` | ReAct 后台线程执行中（含知识库附加 system、定时任务里 `scheduler_context` 触发的同一路径） |
| `team_running` | 团队链 `run_team_turn` 执行中（多槽顺序调用，对用户表现为**同步**一轮） |
| `followup_pending` | 卡片确认等路径触发的**紧随** LLM 调用尚未返回（如 `_chat_style_followup_after_card`） |
| `degraded` | 已知出错且本轮已结束（可选：用于 UI 提示重试，不阻塞新输入） |

> **说明**：`streaming` 与 `react_running` 互斥于「单会话单活跃轮」假设；切换会话时应将非当前会话的活跃相位视为**仅持久化快照或丢弃**（由产品决定）。

### 3.2 子状态（按模式扩展，JSON 可序列化）

统一放入 `state_extension`（或等价字段），便于版本演进。

**Chat / 拟人（`standard`，`mode` 为 `chat` 或 `persona`）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `active_turn_id` | `int` | 与运行时 `turn_id` 对齐 |
| `stream_channel` | `string?` | 可选：`content` / `thinking`（仅调试） |

**ReAct（`standard` 或 `knowledge`，`mode` == `react`）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `react_thread_alive` | `bool` | 是否与 `_react_thread` 一致 |
| `step_index` | `int?` | 若底层暴露步索引则写入；否则可为 `null` |
| `cancel_requested` | `bool` | 是否与 `_react_cancel` 一致 |

**团队（`session_variant` == `team`）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `team_size` | `int` | 自 `SessionMeta.team_size` 复制 |
| `current_slot` | `int?` | 1…`team_size`，执行 `run_team_turn` 循环时更新；结束后置空 |

**知识库（`session_variant` == `knowledge`）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `kb_preset` | `string` | 自 `SessionMeta.kb_preset` 复制，便于审计展示 |

### 3.3 叠加标记（与主相位正交）

可与主相位同时存在（位标志或独立数组）：

- **`memory_extract_busy`**：记忆抽取占用 `llm_busy` 时的旁路标记（不改变用户主相位，但用于调试「为何闲时任务未跑」）。
- **`action_card_pending`**：存在未决 `card`（可从最后一条 assistant 推导，或单独缓存）。

## 4. 各模式状态机（简图）

### 4.1 Chat（含知识库 Chat）

```text
idle --用户发送--> streaming --流结束/中断--> idle
                      | turn.cancelled
                      v
                   idle（带【已中断】消息）
```

### 4.2 拟人（`persona` + `standard`）

与 Chat 相同事件骨架；额外可有 `agent.proactive` 等，**不单独增加顶层相位**，可在 `state_extension` 记 `last_event_type`。

### 4.3 ReAct

```text
idle --用户发送--> react_running --run_react 结束--> idle
        ^                    |
        |                    | cancel / 异常
        +--------------------+
```

### 4.4 团队

```text
idle --用户发送--> team_running（slot 1→…→n）--> idle
```

### 4.5 卡片后续

在用户确认卡片并写入 user 行之后，若触发自动 follow-up：

```text
idle --内部触发--> followup_pending --> idle
```

## 5. 持久化与恢复策略

**文件建议**（与 `meta.json` 同目录）：`dialogue_state.json`（或并入 `meta` 的扩展字段，需版本号 `schema_version`）。

**最小字段**：

```json
{
  "schema_version": 1,
  "session_id": "...",
  "updated_at": "ISO8601",
  "phase": "idle",
  "state_extension": {},
  "last_turn_id": 0,
  "last_error": null
}
```

**写入时机**：相位变化时 debounce 写入（避免每个 `token.delta` 写盘）；至少：`turn.started`、`turn.completed` / `turn.finished`、`react` 线程 `finally`。

**恢复**：`open_session` 时读取；若 `phase` 为 `streaming` / `react_running` / `team_running` 而进程已重启，标记为 **`idle` + `last_error`：会话恢复，上一轮可能未完整结束**（或显示「可继续输入」），**不自动重连流**。

## 6. UI 与调试 API

| API | 作用 |
|-----|------|
| `Api.get_dialogue_state`（Python） | 返回 `{ ok, phase, last_turn_id, llm_busy_depth, last_error }`（与 `dialogue_state.json` 一致；`state_extension` 留待 P2） |
| 前端 | [`web/app.js`](c:/project/golang/agent-ruyi72-desktop-pygui/web/app.js) 订阅 **`state.changed`**，并调用 **`get_dialogue_state`**（`refreshActive` / `open_session`）；ReAct 流式期间对 **`react.start` / `react.progress`** 节流拉取状态以更新 **步序**，**`react.done`** 再拉一次以对齐 `idle`；**`#dialogue-state-bar`** 展示相位 / `state_extension` / `last_error` |

**与现有事件关系**：不替代 `token.delta`；`state.changed` 仅在相位或 `active_turn_id` 变化时发送。

## 7. 与相关机制的关系

- **记忆系统**：长期记忆读写与「对话相位」独立；检索工具调用发生在 `react_running` 或 `streaming` 内部，不在此文档展开 schema。
- **内置定时任务**：触发 `call_llm_once` 或带 `scheduler_context` 的 ReAct 时，`phase` 仍遵循上表，并可在 `state_extension` 增加 `trigger: "scheduled"`。
- **企业级工具流**（见 [企业级Agent工具调用综合方案.md](企业级Agent工具调用综合方案.md)）：若未来接入 L0–R 层，可在 `react_running` 下再嵌 `tool_parse` / `tool_exec` 子状态。
